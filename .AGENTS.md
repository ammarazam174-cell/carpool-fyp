# System Understanding

This document outlines the architecture, tech stack, and key nuances of the Carpool FYP project based on a comprehensive repository analysis.

## 1. Backend Project (`backend/Saffar.Api/`)

### 1.1 Tech Stack 
- **Framework**: .NET 8 Web API
- **Database ORM**: Entity Framework Core 8
- **Database Engine**: SQL Server Express (configured in `appsettings.json` via local connection string)
- **External Integration**: Firebase Admin SDK (used for Push Notifications via FCM)

### 1.2 Architecture & App Structure
- **Controllers**: Exposes endpoints for `Auth`, `Bookings`, `Notifications`, `Rides`, `Users`, and `Vehicles`.
- **DTOs**: Data Transfer Objects structured for mapping incoming and outgoing payload streams (e.g. `CreateRideDto`, `DriverProfileDto`).
- **Models / Entities**: Represents the core EF Core tables (`Booking.cs`, `LoginRequest.cs`, `Rating.cs`, `Ride.cs`, `RideStop.cs`, `User.cs`, `UserNotification.cs`, `Vehicle.cs`).
- **Services**: Contains logic for Firebase authentication & FCM messaging (`FirebaseService.cs`, `PushNotificationService.cs`, `JwtService.cs`).

### 1.3 Key Configurations & Quirks
- **Auth Configuration**: JWT Authentication is currently authored in `Program.cs` and `JwtService`, but the middleware (`app.UseAuthentication()` and `app.UseAuthorization()`) is **commented out** in `Program.cs`. As a result, the API currently allows access to all endpoints without enforcing Bearer token validation at the router level.
- **Ports & CORS**: The server hosts locally on `http://localhost:5000`. CORS is explicitly enabled to allow `[http://localhost:5173]` as a trusted frontend origin.
- **Uploads**: Contains a `wwwroot/uploads` folder implying image storage is done directly to the local disk rather than a cloud bucket.

## 2. Frontend Project (`Frontend/saffar-web/`)

### 2.1 Tech Stack
- **Framework**: React 18, Vite, TypeScript
- **Styling**: TailwindCSS
- **Routing**: React Router DOM (v6)
- **State/Auth Management**: React Context (`AuthContext.tsx`) + LocalStorage
- **Data Fetching**: Axios

### 2.2 Application Structure
Role-based route guarding separates workflows:
- **Public**: Login Page.
- **Driver**: Dashboard, Earnings, Booking Management (Accept/Reject), Ride Management (Create/My Rides/Active), and Profile Setup.
- **Passenger**: Ride Searching, Booking Requests, My Bookings, and Route Details.

### 2.3 Interesting Implementation Details & Bugs Found
- **Bug Discovery (API Port Misalignment)**: The API Axios intercepts and Environment Variables are hardcoded to hit `http://localhost:5128/api`. However, the Backend runs on `http://localhost:5000`. This requires addressing to ensure the two applications can converse.
- **Token Passing**: Interceptors automatically extract the `token` from `localStorage` and tack it onto `Authorization: Bearer <token>` headers on every request, successfully anticipating the activation of JWT guarding.
- **Firebase Messaging**: A `firebase-messaging-sw.js` instance is present in `/public`, working collaboratively with `src/firebase.ts` to coordinate Push Notification Token registration on login/dashboard rendering via vapor keys.
- **Reconstruction History**: `.gitignore` initially blocked standard JSON files resulting in an unversioned `package.json` and `tsconfig.json`. These files were reconstructed dynamically by analyzing typescript imports inside `src/`.
